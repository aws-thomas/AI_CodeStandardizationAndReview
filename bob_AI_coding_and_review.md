# Standardizing Code Quality and Basic Code Review Using AI

## Overview

This document demonstrates how AI coding agents can be used to enforce consistent code quality standards and perform systematic code reviews across a project — tasks that are time-consuming manually and easy to skip under deadline pressure.

The scenario uses **myKB**, a web-based Kanban board application built from scratch, as the working example. The AI tooling used throughout is **IBM Bob**.

Each step includes a **Document this** callout that tells you exactly what to screenshot or copy as a code snippet for your reference documentation (Google Sheets log or single HTML file).

---

## Scenario

> A solo developer is building myKB — a web-based Kanban board with boards, columns, and draggable cards. They are using IBM Bob to write the code. Rather than reviewing each piece of generated code manually as the project grows, they set up AI-driven quality standards and automated review from day one, so the codebase stays consistent as features are added incrementally.

---

## Project Overview: myKB

**myKB** is a lightweight, web-based Kanban board application. Core features:

- Create and manage multiple boards
- Add, rename, and reorder columns (e.g. To Do, In Progress, Done)
- Create, edit, move, and delete cards within columns
- Drag-and-drop cards between columns
- Persist board state across sessions

**Planned stack:**
- Frontend: Next.js with TypeScript
- Backend: FastAPI (Python)
- Database: SQLite
- Styling: Tailwind CSS

## Step-by-Step Guide

### Step 1 — Define your quality standards before writing any code

The first thing to create — before any feature code — is a `BOB.md` file at the project root. This is IBM Bob's source of truth for what "good code" means in this project. Every subsequent IBM Bob session and review agent will read it.

**Create `BOB.md` at project root:**
```markdown
# myKB — IBM Bob Instructions

## Project
Web-based Kanban board. Backend: FastAPI + SQLite. Frontend: Next.js TypeScript.

## Code Quality Standards
- Backend: all database writes must be wrapped in a transaction; never commit partial state
- Backend: never return raw exception messages or stack traces to the client
- Backend: use parameterized queries only — no string-formatted SQL
- Frontend: no `any` types in TypeScript — use explicit interfaces
- Frontend: all API calls must handle loading and error states visibly
- All: write descriptive comments on functions
- Write simple code, do not overengineer, but code readability must be prioritized.
- When writing variables use camel casing.
- All: no commented-out code committed to the repo
```

Having this written down means every IBM Bob coding and review session works from the same bar, regardless of which agent or session writes the code.

> - **HTML file**: place inside a `<pre><code>` block under a "Step 1 — Standards" heading
>
> This is the foundation everything else is measured against, so it must appear first in the doc.

### Step 2 — Use IBM Bob to generate the initial project scaffold

With standards in place, use IBM Bob to create the project structure. Because `BOB.md` is already present, IBM Bob applies the standards from the first line of code it writes.

**Example prompt:**
```
Initialize the myKB project. Create the directory structure described in BOB.md:
a FastAPI backend under backend/ and a Next.js TypeScript frontend under frontend/.
Set up pyproject.toml with uv and package.json with the required dependencies.
Do not write any feature code yet — scaffold only.
```

Starting with a scaffold rather than jumping straight to features gives you a chance to verify the structure before any logic is written.

> **📸 Document this — Screenshot**
> Once IBM Bob finishes, run `tree` (or use the file explorer) to show the generated directory structure, then take a screenshot.
> - **Google Sheets**: paste the screenshot into the "Evidence" column of Step 2's row, with a note: "Scaffold generated by IBM Bob before any feature code"
> - **HTML file**: embed the screenshot as an `<img>` tag under "Step 2 — Scaffold", with a caption describing what IBM Bob created
>
> This screenshot proves the structure was AI-generated from standards, not hand-rolled.

### Step 3 — Use IBM Bob to audit a specific file or module as it is written

As each feature is added, ask IBM Bob to review the new file immediately — before moving on. Be specific about what to check so the output is actionable, not generic.

**Example prompt (after the cards API is written):**
```
Review backend/routers/cards.py against the standards in BOB.md.
Check for: missing transaction boundaries on writes, any raw exception messages
returned to the client, string-formatted SQL, and functions longer than 40 lines.
List findings with line numbers. Do not refactor — just report.
```

IBM Bob reads the file, applies the criteria, and returns a prioritized list of specific issues with line numbers rather than general observations.

> **📸 Document this — Screenshot + Code Snippet**
> Capture two things:
> 1. **Screenshot** of IBM Bob's review response in the terminal/chat, showing the numbered findings with file and line references
> 2. **Code snippet** of the specific flagged code block (copy the offending lines directly from the file)
>
> - **Google Sheets**: screenshot in "Evidence" column, flagged code snippet in "Code Before Fix" column, both in Step 3's row
> - **HTML file**: screenshot under "Step 3 — File Review", followed by a `<pre><code>` block of the flagged code, labelled "Code flagged by IBM Bob"
>
> Pairing the AI's finding with the actual code makes the review evidence concrete and verifiable.

### Step 4 — Use a sub-agent to review across all files at a milestone

At natural checkpoints (end of a sprint, before a feature merge), delegate a full-codebase sweep to a dedicated IBM Bob review agent. This is more thorough than interactive review and produces a structured report.

**Create `.bob/agents/code-quality.md`:**
```markdown
---
name: code-quality
description: audit code quality across the myKB codebase
tools: Read, Glob, Grep, Write
---

Read all Python files under backend/ and all TypeScript files under frontend/src/
using Glob and Read. Check each file against the standards in BOB.md.
Write a structured report to planning/CODE_QUALITY_REPORT.md with:
- File path
- Issue description
- Line number
- Severity (high / medium / low)
Do not modify any files. Report only.
```

**Invoke it with:**
```
Use the code-quality sub-agent to audit the full codebase.
```

The agent works through every file systematically and produces a single report the developer can triage.

> **📸 Document this — Screenshot + Code Snippet**
> Capture two things:
> 1. **Screenshot** of IBM Bob launching the sub-agent and confirming it has written the report
> 2. **Code snippet** of the relevant section of `planning/CODE_QUALITY_REPORT.md` — specifically any high-severity findings
>
> - **Google Sheets**: screenshot in "Evidence" column, high-severity findings as a code snippet in "Findings" column, Step 4's row
> - **HTML file**: screenshot under "Step 4 — Sub-Agent Sweep", then a `<pre><code>` block with the report extract, labelled "CODE_QUALITY_REPORT.md (high severity items)"
>
> The report extract is the audit trail — it shows what the AI found before any human touched the code.

### Step 5 — Automate the review on every IBM Bob session stop

Use an IBM Bob Stop hook to run a lightweight review automatically after every AI coding session. This catches regressions as they are introduced rather than at milestone reviews.

**In `.bob/settings.json`:**
```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bob exec \"Review changes since last commit for code quality issues against the standards in BOB.md and append findings to planning/REVIEW.md\""
          }
        ]
      }
    ]
  }
}
```

Every time IBM Bob finishes a task, the hook fires, reviews only the diff since the last commit, and findings are appended to `planning/REVIEW.md` — no manual step required.

> **📋 Document this — Code Snippet**
> Copy the full `.bob/settings.json` hook configuration as a code snippet.
> - **Google Sheets**: paste into the "Standards / Config" column of Step 5's row, with a note: "Hook runs automatically — no manual trigger needed"
> - **HTML file**: place inside a `<pre><code class="language-json">` block under "Step 5 — Automated Hook", with a short explanation of when it fires
>
> Showing the hook config is important — it demonstrates that review is baked into the workflow, not optional.

### Step 6 — Use IBM Bob to apply fixes, not just find them

Once issues are reported, use IBM Bob to fix them systematically. Give it the report and ask it to apply one category of fix at a time — for example, adding missing transactions to all card write endpoints.

**Example prompt:**
```
planning/CODE_QUALITY_REPORT.md lists 3 high-severity issues: missing transaction
boundaries in backend/routers/cards.py at lines 42, 78, and 103.
Wrap each of those database write sequences in a try/except with a rollback,
following the pattern already used in backend/routers/boards.py line 31.
Do not change any other logic.
```

Pointing IBM Bob at an existing correct pattern ("use the same approach as line 31") prevents it from inventing its own style and keeps the codebase consistent across files.

> **📋 Document this — Before & After Code Snippets**
> Copy two code snippets side by side:
> 1. **Before**: the original flagged code block from `cards.py` (lines 42–50 or so)
> 2. **After**: the same block after IBM Bob applied the fix
>
> - **Google Sheets**: "Code Before Fix" column and "Code After Fix" column in Step 6's row
> - **HTML file**: two `<pre><code>` blocks side by side (use a two-column `<div>` or a table), labelled "Before" and "After", under "Step 6 — Fix Applied"
>
> Before/after snippets are the clearest possible evidence that AI-driven fixing works — they show the exact change, not just a description of it.

### Step 7 — Verify the fixes with a second IBM Bob pass

After fixes are applied, run the review agent on the changed files to confirm the issues are resolved — before committing.

**Example prompt:**
```
Re-run the code-quality check on backend/routers/cards.py only.
The previous report flagged missing transactions at lines 42, 78, and 103.
Confirm whether those are now resolved. Report pass/fail per issue.
```

This closes the loop without requiring a human to re-read every file.

> **📸 Document this — Screenshot**
> Take a screenshot of IBM Bob's pass/fail verification response, clearly showing each previously flagged issue marked as resolved.
> - **Google Sheets**: screenshot in "Evidence" column of Step 7's row, with a note: "All high-severity issues resolved — verified by IBM Bob before commit"
> - **HTML file**: embed as an `<img>` under "Step 7 — Verification", with a caption listing the issue count and outcome (e.g. "3/3 issues resolved")
>
> This screenshot is the sign-off. Without it, the documentation shows findings but not resolution.

## Key Principles

- **Standards first, code second.** Write `BOB.md` before any feature code — IBM Bob will follow it from the first line it generates.
- **Specificity beats generality.** "Check for missing transactions in cards.py" produces better output than "review this file."
- **Separate finding from fixing.** Run one IBM Bob pass to report, a second to fix. Mixing them leads to unrequested refactors.
- **Give IBM Bob a pattern to follow.** Point it at existing correct code rather than letting it invent a style.
- **Automate the repetitive parts.** Stop hooks and sub-agents handle routine checks so human review time is spent on judgment calls, not mechanical scanning.
- **Capture as you go.** Take screenshots and copy snippets immediately after each step — reconstructing evidence after the fact is unreliable.
